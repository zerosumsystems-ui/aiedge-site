# Quant beta-test report — chart + bars APIs

**Run:** 2026-05-14 ~12:30 UTC against `https://www.aiedge.trade`
**Method:** direct HTTP to `/api/bars`, `/api/bars/live`, `/api/live-bars/diagnostics`; UI driven via Playwright MCP.
**Mindset:** "what would a quant relying on this data discover that breaks their workflow?"

---

## Passed cleanly

- **OHLC integrity.** 9 symbols (SPY, NVDA, AAPL, TSLA, MSFT, AMZN, GOOGL, META, QQQ) on 2026-05-13 RTH 5min: 78 bars each, **0 invariant violations** (`high >= max(o,c)`, `low <= min(o,c)`, `high >= low`, `volume >= 0`).
- **Reproducibility.** Same query twice produced identical bytes (0 diffs).
- **1m → 5m aggregation accuracy.** When the full 390 1m bars are fetched (`limit=400`), aggregating them to 5m matches the 5m endpoint exactly — 78/78 bars match on (o,h,l,c,v).
- **Live bars dedup.** `/api/bars/live?ticker=SPY` returned 10 identical bars across three snapshots ~3s apart. No duplicate timestamps, no OHLC violations, no values mutating for past timestamps.
- **Aggregator health.** `/api/live-bars/diagnostics` reports `ok:true`, Fly app reachable, 91ms latency, `healthy:true` (the post-close flap I saw yesterday in `/fly status` has resolved during the active session).

## Bugs found

### 1. Silent 78-bar default limit on `/api/bars` — **HIGH**

**Repro:**
```
GET /api/bars?ticker=SPY&from=2026-05-13&to=2026-05-13&session=rth&tf=1min
→ 78 bars (14:42 → 15:59 ET). Caller expected 390.

GET /api/bars?ticker=SPY&from=2024-01-01&to=2026-05-13
→ 78 daily bars (2026-01-22 → 2026-05-13). Caller expected ~600.
```

Default response is truncated to 78 bars regardless of timeframe, with no warning. Anyone using `/api/bars` as a data source for backtests, indicator math, or feature extraction will silently get a 78-bar tail of their requested range. Workaround is `&limit=...` but it's undocumented in the response.

**Suggested fix:** either return a `truncated: true` field + suggested next-page params, or default to a generous cap and document it.

---

### 2. `tf=30min` silently returns 5min bars — **MEDIUM**

**Repro:**
```
GET /api/bars?ticker=SPY&from=2026-05-13&to=2026-05-13&tf=30min&session=rth
→ { "timeframe": "30min", "effectiveTimeframe": "5min", "bars": [78 bars with 300s deltas] }
```

Metadata is internally contradictory: `timeframe=30min` but `effectiveTimeframe=5min` and the actual data deltas are 300s. A caller trusting `timeframe` would mis-bin the data.

**Suggested fix:** either implement 30min aggregation, or reject `tf=30min` with a 400 listing supported timeframes.

---

### 3. 502 on `tf=1h` / `tf=4h` for 5–14-day ranges that touch recent data — **HIGH**

**Repro:**
```
GET /api/bars?ticker=SPY&from=2026-05-01&to=2026-05-13
→ 502 Bad Gateway
  body: {"error":"databento 422: {\"detail\":{\"case\":\"data_schema_not_fully_available\",
                                              \"message\":\"Schema `ohlcv-1h` for d..."}}
```

`pickSchema()` returns `ohlcv-1h` for 4–14-day ranges, but Databento doesn't have hourly data published for the most recent window (similar to the existing 30-min `DATABENTO_FEED_LAG_MS` constant used for 1m). The 422 from upstream leaks as a 502, and auto-mode is unusable for the most common "last two weeks" range.

**Suggested fix:** detect `data_schema_not_fully_available`, fall back to aggregating 1m bars (or daily, depending on length). Or honor a `DATABENTO_1H_LAG_MS` similar to the existing 1m constant and clip the upper bound of the request.

---

### 4. `?symbol=` URL param is silently ignored — **MEDIUM**

**Repro:** open `https://www.aiedge.trade/chart?symbol=NVDA` → chart shows SPY ($742.30), not NVDA.

`src/app/chart/page.tsx` doesn't read `searchParams` and `ChartClient` doesn't either; the route boots whatever the persisted/default symbol is. Deep-linking from external tools (Slack, Linear, watchlists, alerts) doesn't work.

**Suggested fix:** in `ChartClient`, read `useSearchParams().get('symbol')` and seed the symbol store with it on mount when present. Should respect the "symbol stickiness" convention from the chart skill — once seeded, live updates don't yank.

---

### 5. Invalid ticker returns 200 with empty `bars` instead of 404 — **MEDIUM**

**Repro:**
```
GET /api/bars?ticker=ASDFASDF&from=2026-05-13&to=2026-05-13
→ 200 OK, { "bars": [], "source": "databento", ... }
```

A quant pipeline calling this with a typo (`SPY` → `SYP`) gets `bars: []` and may interpret it as "no trading activity that day" rather than "ticker doesn't exist." Indistinguishable from a real empty result (e.g., a holiday with `session=rth`).

**Suggested fix:** Databento returns a recognizable error for unknown tickers — forward as 404 with `{ "error": "unknown ticker" }`. Real empty windows can stay 200 + `bars: []`.

---

### 6. Future date returns 502, weekend returns 200 empty — **LOW**

```
GET /api/bars?ticker=SPY&from=2027-01-01&to=2027-01-01
→ 502 Bad Gateway

GET /api/bars?ticker=SPY&from=2026-05-09&to=2026-05-09   (Saturday)
→ 200, { "bars": [] }
```

Inconsistency: weekends/holidays return 200 empty, but future dates 502. Both are "no data" cases conceptually. Should normalize to 400 (`from > today`) or 200 empty.
